Step-by-step: what's used and why

1

Reading uploaded files

Tools: pypdf, python-docx, python-pptx
Why: each file type (PDF, Word, PowerPoint) is a different format under the hood, so a dedicated reader is used for each one to pull the text out cleanly.
2

Reading scanned pages and photos

Tools: EasyOCR (main), Tesseract (backup)
Why: a scanned page or photo has no real "text" in it — it's just an image. OCR looks at the picture and figures out what the words say. Two engines are used so that if one struggles with a tricky image, the other can still get the job done.
3

Breaking text into bite-sized pieces

Tool: custom chunker (uses tiktoken when available)
Why: the AI can only "think about" a limited amount of text at once, so long documents are split into overlapping pieces small enough to search and reason about accurately.
4

Turning text into something searchable by meaning

Tool: Ollama running the nomic-embed-text model
Why: this converts each chunk of text into a kind of "meaning fingerprint," so later the app can find passages that match the idea behind your question — not just matching words.
5

Storing document knowledge

Tool: ChromaDB
Why: it's a database built specifically to store those "meaning fingerprints" and quickly find the closest matches to a new question.
6

Storing spreadsheets and numbers

Tool: DuckDB (loaded via pandas)
Why: numbers need to be added up exactly, not guessed at by an AI. DuckDB is a real database that can calculate totals, averages, and counts with total accuracy.
7

Figuring out what kind of question you're asking

Tool: custom rule-based router + Ollama as a fallback judge
Why: a document question and a "what's the total sales" question need completely different handling. Simple word patterns catch most cases instantly; the AI only gets asked to decide when it's genuinely unclear, which keeps things fast.
8

Finding the right passages (document questions)

Tools: ChromaDB (meaning search) + BM25 (keyword search), combined, then a CrossEncoder re-ranker
Why: meaning-based search alone can miss exact terms (like product codes or names), and keyword search alone can miss paraphrased ideas. Using both and then double-checking the top results with a more careful re-ranker gives more reliable answers.
9

Answering number questions

Tool: AI-generated SQL, checked, then run in DuckDB
Why: the AI writes the calculation as a real database query instead of doing the math "in its head," so the answer is guaranteed correct. The query is also checked to make sure it can only read data, never change or delete it.
10

Writing the final answer

Tool: Ollama running Llama 3.2 (all running on your own machine)
Why: this is the only part that "writes like a person" — everything before it just gathered the right facts. Running it locally means nothing ever has to leave your network.
11

Remembering the conversation

Tool: SQLite
Why: a lightweight, reliable way to store your chat history per session, so follow-up questions like "what about him?" still make sense.
12

Reading answers aloud

Tool: Piper (natural-sounding voice), falls back to pyttsx3
Why: Piper sounds more natural but needs a voice file installed; pyttsx3 is a built-in backup so the feature always works even before that's set up.
13

Exporting an answer as a PDF

Tools: server-side PDF builder + jsPDF in the browser
Why: lets you save or share a clean, formatted copy of an answer without needing an internet connection or third-party service.
14

Showing charts for number questions

Tool: ECharts
Why: some answers (totals by department, trends over time) are easier to understand as a picture than as a sentence.
15

Running the whole app

Tools: FastAPI (the engine that serves everything), Docker (optional, for easy setup)
Why: FastAPI is what actually listens for your requests and routes them to the right piece above. Docker just makes it easy to package the whole thing (app + AI model) to run on any machine without a complicated manual setup.
